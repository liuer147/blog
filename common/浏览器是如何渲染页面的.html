    浏览器中页面的渲染分为两个阶段：网络请求阶段 和 页面渲染阶段。

    ...

    # 页面渲染阶段

    当浏览器的 网络线程 收到 HTML 文档后，会产生一个渲染任务，并将其传递给 渲染主线程 的消息队列。 在 事件循环机制 的作用下，渲染主线程 取出 消息队列 中的 渲染任务，开启渲染流程。 整个页面的渲染流程分为多个阶段，分别是：解析 HTML、样式计算、布局、分层、绘制、分块、光栅化、合成。

    # 解析 HTML

    解析过程中遇到 CSS 解析 CSS，遇到 JavaScript 执行 JavaScript。 为了提高解析效率，浏览器在开始解析前，会启动一个 预解析的线程，率先下载 HTML 中的**外部 CSS 文件** 和**外部的 JavaScript 文件**。

    **如果主线程解析到 ****link**标签位置，此时外部的 CSS 文件还没有下载解析好，主线程不会等待，继续解析后续的 HTML。这是因为 下载和解析 CSS 的工作是在 **_预解析线程 _**中进行的。这就是 CSS 不会阻塞 HTML 解析的根本原因。

    如果主线程解析到 **script** 标签位置，会停止解析 HTML，转而等到 JavaScript 文件下载好，并将其全局代码解析并执行完成后，才能继续解析 HTML 。这是因为 JavaScript 代码的执行过程可能会修改当前的 DOM 树，所以 DOM 树的生成必须暂停。这就是 JavaScript 会阻塞 HTML 解析的根本原因。

    这一步完成后，会得到 DOM 树和 CSSOM 树，浏览器的 默认样式、内部样式、外部样式、行内样式 均会包含在 CSSOM 树中。

    # 样式计算

    主线程会遍历得到的 DOM 树，依次为树中每个节点计算出它最终的样式，称之为 Computed Style。

    在这一过程中，很多 预设值 会变成 _绝对值 _。比如 color: red;会变成color: rgb(255, 0, 0);相对单位会变成绝对单位，比如font-size: 2em;会变成font-size: 24px;

    灵魂一问：为什么 head元素是隐藏的？ 因为浏览器存在默认样式表。在默认样式中，head元素被设置了display: none;。同样的，为什么会有块级元素？因为默认情况下设置了display:block;

    这一步计算后，会得到一颗 _带有样式 _的 DOM 树。

    # 布局

    布局阶段会依次遍历 DOM 树的每一个节点，计算每个节点的 _几何信息 _。例如节点的宽高、相对包含块的位置。 大部分时候，DOM 树和布局树并非一一对应。比如：

    • display: none;的节点没有几何信息，因此不会生成到布局树；
    • 使用了伪元素选择器，虽然 DOM 树中不存在这些伪元素节点，但它们拥有几何信息，所以会生成到布局树中。
    • 还有 匿名行盒、**匿名块盒 **等都会导致 DOM 数和布局树无法一一对应。
      • HTML元素里的 _文本内容 必须在 行盒 中 _，
      • _行盒 和 块盒 不能相邻 _

    如图：

    1. 布局树中，因为 块元素<p>中直接存在文本 a、c，所以在生成布局树时，会使用 匿名行盒包裹内容a、c。
    2. 文本内容 b直接与块盒<p>相邻，所以应该使用 **匿名块盒 包裹b，但由于文本内容不能直接和块盒相邻，此时还会生成 行盒 包裹b。 （图中是错的。文本**b**的分支应该是：**div** -> **匿名块盒** -> **匿名行盒** -> ****b**）

    # 分层

    主线程会使用一套复杂的策略对整个布局树中进行 分层。

    分层的好处：将来某一层改变后，仅会对该层进行后续处理，从而提升效率。

    滚动条（滚动条单独分一层）、堆叠上下文、transform、opacity 等样式或多或少的影响分层； 可以通过will-change属性更大程度影响分层结果。 注意：不要分层过多。

    will-change (opens new window) 用于告诉浏览器哪些属性会发生变化。

    # 绘制

    主线程 会为 每个层 单独 产生绘制指令集，用于描述这一层的内如该如何画出来。

    # 分块

    合成线程 也在 渲染进程中。

    主线程将每个图层的绘制信息提交给合成线程。 合成线程 首先对 **每个图层 **进行 分块，将其划分为更多的小区域。它会从 **线程池 **中拿取多个线程来完成分块工作。

    # 光栅化

    合成线程 会将 块信息 交给 GPU 进程。 GPU 进程 会开启多个线程来完成 光栅化，并**_优先处理靠近视口区域的块 _**。 光栅化的结果，就是一块一块的位图。

    # 画

    合成线程 拿到每一个层、每一个块的位图后，生成一个个【指引quad】信息。 指引会标识出每个位图应该画到屏幕的哪个位置，以及会考虑到旋转、缩放等变形。

    所以，**transform**发生在合成线程！与渲染主线程无关，这就是它效率高的本质原因。

    合成线程会把quad提交给 GPU进程，由 GPU进程 产生系统调用，提交给 GPU硬件，完成最终的屏幕成像。

    为什么 合成线程 需要把 quad 提交给 GPU进程产生系统调用呢？ 因为浏览器的渲染进程是一个沙盒，它与计算机操作系统不相关。因此它是安全的，当浏览网站中病毒时，没法影响操作系统。同样的，他没法直接调用操作系统的接口。所以会将执行信息交给GPU 进程来产生系统调用。

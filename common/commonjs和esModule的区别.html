    # 背景

    早期 JavaScript 开发很容易存在 全局污染 和 依赖管理混乱 的问题。

    <body>
  <script src="./f2.js"></script>
  <script src="./f1.js"></script>
</body>

    • 全局污染

    当 HTML 文件中这样引入 JavaScript 脚本时，如果f1.js中存在和f2.js中同名的变量，那么对于后续 JavaScript 脚本来说，则f1.js中的变量会覆盖f2.js中的变量。如果这个后续插入的脚本是不同的开发人员编写的就很容易引发问题。

    • 依赖管理混乱

    对于这种引入的 JavaScript 脚本来说，下方的 JS 脚本可以访问 上方 JS 脚本声明的变量，而上方的 JS 脚本却无法访问 下方的 JS脚本中的内容。这就导致开发者还需要花费心智来管理这个script脚本的顺序。

    # esModule

    与<script type="module">关联或者通过import语句加载的 JavaScript 文件会被认定为模块。 所有模块都会像<script defer>加载的脚本一样按顺序执行。解析到<script type="module" >标签后会立即下载模块文件，但执行会延迟到文档解析完。

    <!-- 第二个执行 -->
<script type="module"></script>

<!-- 第三个执行 -->
<script type="module"></script>

<!-- 第一个执行 -->
<script></script>

    支持 esModule 的浏览器加载模块的流程：浏览器会先通过网络请求入口模块对应的 JavaScript 文件，然后解析它是否有import其他模块，如果存在import其他模块，则依次按需通过网络请求对应的 JavaScript 文件。这个过程是浏览器从顶级模块加载整个依赖图的过程，这个过程是异步的。等到解析完依赖图之后，应用程序就可以正式加载模块了。 esModule 的基本导出、导入语法如下：

    // 模块导出内容
export const n1 = 'xxx'

export default 'xxx'

const n2 = 'n2'
export {
  n2
}

// 模块导入内容
import xx from 'xx.js'
import { xx } from 'xx.js'

    # commonjs 和 esModule 的区别

    相同点：

    1. esModule 和 commonjs 都会对加载过的模块进行缓存。无论import/require多少次，其模块代码只会执行一次。因此可以解决模块循环引用的问题。

    不同点：

    1. 在 CommonJS 中，模块加载执行是同步操作（运行时才处理），也就是说require()语句可以放在类似条件语句中；而 ES Module 则是在预处理阶段解析依赖图，解析完依赖图之后才会执行（加载）模块代码。import/export语句会被提升到模块顶部（因此其语句在模块中的哪个位置不重要），所以import/export语句不能放到类似条件语句块中。
    2. ES Module 模块默认在严格模式下执行；而 CommonJS 不是。
    3. ES Module 模块导入的值是只读的；而 CommonJS 不是。
    4. ES Module 模块导出的是原始值的引用（可以理解为普通值也是引用）；而 CommonJS 导出的是原始值的拷贝。

    # commonjs 导出，commonjs 引入

    webpack 会声明一个模块对象__webpack_modules__，其保存着依赖图中每个有导出内容的模块。模块相对根目录的相对路径作为属性 key，文件内容包装成函数作为值。 webpack 会声明一个__webpack_require__函数，作为有导入内容的模块中require的实现。它会给每个模块都声明一个包含exports属性的对象{ exports: xxx }作为模块对象。并且返回的是此模块对象的exports属性值

    var __webpack_modules__ = {
  "./src/title.js": (module) => {
    const title = '我是title'
    module.exports = title
  }
};

var __webpack_module_cache__ = {};
function __webpack_require__(moduleId) {
  var cachedModule = __webpack_module_cache__[moduleId];
  if (cachedModule !== undefined) {
    return cachedModule.exports;
  }
  var module = __webpack_module_cache__[moduleId] = {
    exports: {}
  };

  __webpack_modules__[moduleId](module, module.exports, __webpack_require__);

  return module.exports;
}

    # esModule 导出，commonjs引入

    如果是 esModule 形式的导出的话，webpack 会给__webpack_require__中声明且实际返回的module.exports定义[Symbol.toStringtag]: 'Module'以及_esModule: true以作为区分，并且会将默认导出形式的内容export default xxx作为module.exports对象的default属性。 针对所有的 esModule 导出呢，实际上都会变成module.exports对象上的get()访问器属性，且访问器直接返回就是模块中声明的对应变量值。这么做有什么好处呢？

    1. 访问器属性只定义了getter，所以它对于 导入内容的模块 是只读的。
    2. getter能够实时的获取定义的变量的最新值，而 **_不会存在值复制所导致的不及时更新 _**的情况。

    # 那么，title.js是怎么被加载使用的呢？

    webpack 加载 chunk 代码的大致流程为：__webpack_require__.e -> __webpack_require__.f.j -> __webpack_require__.l。 在__webpack_require.l中使用了 jsonp 的方式请求了dist/src_title_js.js，插入到了document.header，不过，webpack 在script标签加载完成onload的回调函数中删除了这个插入的script标签。 并且，在__webpack_require.l函数中还会先获取文档中所有的script标签，判断是否已经存在src属性与要加载的 chunkId（对应文件的路径或名字）是否相同，或者data-webpack属性是否与对应的webpack-test:chunk-${chunkId}，如果已经相同则表示被文档引入了，则直接延时执行对应的执行完成回调函数（此回调中会删除该script标签，且清除延时器）。

    # dist/src_title_js.js中的代码是怎么执行的呢？

    在dist/src_title_js.js中调用了全局对象属性webpackChunkwebpack_test的push方法，且将自己及自己依赖的模块相关内容传入（模块第一次加载时，会执行模块代码）。那么在main.js中也是通过添加全局属性webpackChunkwebpack_test，且重写了push方法开执行副作用。将dist/src_title_js.js中的模块内容集合，合并到main.js中的模块内容集合__webpack_modules__中。

    # 那么，什么时候会执行对应的 Promise 实例的 resolve 回调呢？

    在全局对象属性webpackChunkwebpack_test重写的push方法中会执行对应的 resolve 回调（执行完之后会清除状态）。因为当脚本script加载成功后就会自动执行一次目标模块中的代码，从而就会执行重写的push方法。 等到全部的模块执行完之后，再在__webpack_require__.e().then中调用__webpack_require__执行目标 模块的代码（注意，模块中的代码在开始dist/src_title_js.js是放入到函数中，并没有执行），并且再次链式调用import()的回调代码，中间的那层.then同样保证的此处的import()回调能访问到目标模块导出的内容（__webpack_require__函数会返回module.exports，其作为下一次then调用的数据）。

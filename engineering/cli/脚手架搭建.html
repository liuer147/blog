    1、初始化项目

    # 1、初始化项目

    创建一个文件夹，执行npm init -y命令，此命令会以默认配置创建一个package.json文件。如不想使用默认配置，则执行npm init，它会交互式的让用户填写相应配置。 有了package.json之后，我们就可以基于这个文件来管理当前项目：诸如依赖，项目名称，版本号等项目信息。

    # 2、创建js文件

    在js文件中写入下面代码

    #!/usr/bin/env node

console.log('Hello World!')

    # #!/usr/bin/env node这行代码有什么用呢？

    **#!/usr/bin/env node**这行代码是告诉系统在命令行访问此文件时，会在系统环境变量中寻找**node**解释器来执行此文件

    #!用于指定这个脚本文件的解释程序，它通常在Linux或Unix中脚本文件的第一行出现，被称为**Shebang**。 **/usr/bin/env node**是告诉系统可以在**$PATH**中查找**node****解释器，**不同用户不同的脚本解释器可能安装在不同的目录下。而不管脚本解释器安装在哪个目录下，在全局可以执行的话总归是要将其可执行文件的目录放到系统PATH环境变量中。从而可以解决各个用户的各个脚本解释器路径不同的问题。

    此时，我们已经可以尝试在git bash中访问(运行)此js文件了。但这还不够，一直通过文件路径访问此js文件会显得非常冗余。我们需要像vue-cli那样，通过简单执行即可执行(访问)相应文件。

    # 为什么要添加bin字段呢？

    npm官网对bin的描述 (opens new window) bin字段是**命令名到本地文件名的映射。**当我们使用npm来安装包时，如果该包的package.json中存在bin字段，就会在node对应的环境变量(node可执行文件路径)下创建与**bin**字段属性相同名称的**windows**命令脚本。此时，我们在命令行界面输入**windows**命令脚本的名称，就可以执行对应的文件了。如果这个对应的文件不是以#!/usr/bin/env node开头，那么它将不会执行。

    # 对于本地项目来说，添加bin字段之后，我们怎样操作才能像上面那样执行对应的命令呢？

    我们可以执行**npm link**将命令挂载到全局，**npm unlink**取消挂载

    • npm link your-lib: 将当前项目中node_modules下指定的库文件链接到node全局 node_modules下的库文件
    • npm link: 将当前项目链接到node全局node_modules中作为一个库文件，并解析bin配置创建可执行文件，添加到系统环境变量中。
    • npm unlink: 删除当前项目对应的全局node_modules软链接，并删除环境变量中的可执行文件。
    • npm unlink your-lib: 删除当前项目对应的全局node_modules软链接，以及删除当前项目对应的your-lib依赖

    此时，我们已经可以通过执行命令来执行对应的文件了。

    # 4、解析命令行参数

    目前为止，我们的脚手架只能打印Hello World!，这还远远不够！我们需要让它捕获用户的输入，并作出不同的执行逻辑。 想要捕获用户的输入，我们可以借助nodejs中全局对象的argv属性。

    process.argv会返回一个数组。数组前两项是固定的：第一项为**node.exe**的所在路径；第二项是打印**process.argv****文件的所在路径。**剩余的为用户的输入。 现在，我们可以对用户的输入进行处理了。 接下来，我们修改代码，让脚手架能做以下处理：

    • 当用户输入-V, --version时，我们打印package.json中的版本号(version属性)。
    • 当用户输入-H, --help时，我们可以打印脚手架的帮助信息。
    • 当用户输入ls时，我们打印当前目录的普通文件(文件名不是以.开头)
    • 当用户输入ls -a时，我们打印当前目录的所有文件
    #!/usr/bin/env node
const fs = require('fs')

const argv = process.argv

const [,,arg1, arg2] = argv

switch (arg1) {
    case '-H':
    case '--help':
        console.log('Usage: hx <...args>')
        console.log('args:')
        console.log('\t-H, --help\t\t打印帮助信息')
        console.log('\t-V, --version\t\t打印项目版本号')
        console.log('\tls\t\t打印当前目录普通文件')
        console.log('\tls -a\t\t打印当前目录所有文件')
        break
    case '-V':
    case '--version':
        console.log(require('./package.json').version)
        break
    case 'ls':
        const files = fs.readdirSync(process.cwd())
        if (arg2 == '-a') {
            console.log(files.join(' '))
        } else {
            console.log(files.filter(item => !item.startsWith('.')).join(' '))
        }
}

    发现已经成功的对用户的输入进行了处理。 但是，就目前为止还有很多弊端。每当我们增加一个指令时，都需要去手动修改与--help相关操作；脚手架还没有提供获取"指令帮助信息"的指令，如果提供的话，每一个指令都需要在逻辑操作代码中去嵌入获取帮助信息的操作。无疑，这是重复而又烦恼的。
